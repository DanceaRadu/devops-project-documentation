\section{Introduction and Task Description}

Modern applications require continuous monitoring in order to ensure reliability, performance and stability. Monitoring 
systems provide visibility into the current state of the infrastructure and the application itself, while alerting systems enable 
administrators to react quickly when abnormal behavior occurs.

The goal of this practical assignment was to instrument an existing application with a monitoring and alerting stack using industry-standard 
tools. The implemented solution includes:

\begin{itemize}
    \item \textbf{Prometheus} for metrics collection and storage
    \item \textbf{Grafana} for metrics visualization and alerting
    \item \textbf{Loki + Promtail} for centralized log aggregation
    \item \textbf{Spring Boot Actuator + Micrometer} for application metrics exposure
    \item \textbf{Node Exporter} for host-level hardware and operating system metrics
\end{itemize}

The monitored application is a Spring Boot backend running inside a Docker container on an Unraid server. The monitoring stack 
was designed to provide visibility into:

\begin{itemize}
    \item Host machine resource utilization
    \item Application health and availability
    \item JVM statistics and memory usage
    \item HTTP request statistics and backend errors
    \item Reverse proxy and container logs
\end{itemize}

Additionally, multiple alerting rules were configured in Grafana in order to detect critical conditions such as high CPU usage, excessive 
memory usage, low disk space, backend outages, and HTTP 5xx server errors. Notifications were configured using Discord webhooks and SMTP integration.

The final result is a centralized observability platform capable of monitoring both infrastructure and application-level behavior in real time.

\section{The monitoring stack}

\subsection{Spring Boot Actuator and Micrometer}

In order to monitor the backend application, I integrated Spring Boot Actuator together with Micrometer Prometheus support.

Spring Boot Actuator provides monitoring and management endpoints that expose useful information about the internal state of the application. 
By default, Actuator exposes endpoints related to health checks, metrics, application information and runtime statistics.

Micrometer acts as an abstraction layer for application metrics and automatically exposes them in a format compatible with Prometheus.

For security reasons, the Actuator endpoints were configured to run on a dedicated internal management port instead of the main application port. 
This ensures that monitoring-related endpoints are accessible only internally through Docker networking and are not exposed publicly.

The following metrics are automatically collected and exposed:

\begin{itemize}
    \item JVM heap and memory usage
    \item JVM thread statistics
    \item HTTP request count
    \item HTTP request latency
    \item HTTP 5xx error rates
\end{itemize}

These metrics are periodically scraped by Prometheus and later visualized inside Grafana dashboards.

\subsection{Monitoring Stack Architecture}

The monitoring infrastructure was deployed entirely using Docker containers. All monitoring-related services were grouped inside a dedicated 
Docker Compose stack in order to simplify deployment, maintenance and service communication.

A shared external Docker network named \texttt{beamful-monitoring} was created so that services running in separate Docker Compose stacks 
could communicate internally without exposing additional ports publicly.

Additionally, the monitored Spring Boot backend container was also attached to the same shared Docker network, allowing Prometheus to scrape 
application metrics directly through Docker DNS resolution.

\subsection{Prometheus}
Prometheus is the core metrics collection component of the monitoring stack. Its primary role is to periodically scrape metrics from configured 
targets and store them in a time-series database.

Prometheus was set up with a global scrape interval of 10 seconds and the following targets were configured:

\begin{itemize}
    \item Node Exporter
    \item Spring Boot Actuator
    \item Prometheus internal metrics
\end{itemize}

\subsection{Grafana}

Grafana is responsible for visualization and alert management. It connects directly to Prometheus and Loki as data sources and provides dashboards for infrastructure monitoring, application metrics and centralized logging.

Grafana was also configured to handle alerting rules and notifications using Discord webhooks and SMTP email notifications, but more on this later.

The dashboards were designed to provide a centralized overview of both the host machine and the monitored backend application.

\subsection{Node Exporter}

Node Exporter is used to expose host-level hardware and operating system metrics from the Unraid server. The exporter collects metrics such as:

\begin{itemize}
    \item CPU usage
    \item RAM usage
    \item Disk usage
    \item Filesystem statistics
    \item Network traffic
\end{itemize}

These metrics are scraped directly by Prometheus and visualized inside Grafana dashboards.

\subsection{Loki}

Loki is the centralized log aggregation system used in this project. Unlike traditional logging systems that fully index log contents, 
Loki indexes only metadata labels while storing the raw logs separately. This significantly reduces storage usage and improves 
efficiency for small and medium-sized deployments.

Loki receives logs from Promtail and stores them for later querying inside Grafana. The following types of logs were collected:

\begin{itemize}
    \item Docker container logs
    \item Nginx reverse proxy access logs
    \item Nginx reverse proxy error logs
\end{itemize}

\subsection{Promtail}

Promtail is responsible for collecting logs and forwarding them to Loki. Promtail was configured to monitor both Docker container 
logs and file-based logs generated by Nginx.

The configuration also included automatic extraction of proxy host identifiers from Nginx log filenames using regular expressions. 
This allowed filtering logs individually for each reverse proxy host directly from Grafana.

\section{The dashboards}

Several dashboards were created in Grafana in order to monitor both infrastructure-level and application-level behavior. The created dashboards are as follows:

\subsection{Infrastructure Dashboard}

The infrastructure dashboard focuses on host-level metrics collected through Node Exporter. The dashboard contains panels for:

\begin{itemize}
    \item CPU utilization
    \item RAM usage
    \item Available disk space
    \item Network traffic
\end{itemize}

Threshold coloring was configured for multiple panels in order to visually indicate abnormal and high values. For example, the 
CPU graph turns red when the CPU utilization exceeds 90\%.

This dashboard is useful for quickly identifying issues related to container crashes, application errors and reverse proxy problems.
